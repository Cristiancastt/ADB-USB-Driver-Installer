## Why?

Setting up ADB on a new machine is always the same story — find Google's platform-tools page,
download the right zip, extract it somewhere, add it to PATH, and on Windows you probably also
need USB drivers. It's not hard, but it's annoying.

Also the other tools to set up ADB are either outdated, Windows-only, or the binaries are sketchy... 😬

This tool does all of that in one shot. Run it, follow the wizard, done.

## Features

- **Cross-platform** — Works on Windows, Linux, and macOS
- **Interactive wizard** — Step-by-step guided installation
- **USB drivers** — Automatically handles Windows USB driver installation
- **PATH setup** — Automatically adds adb/fastboot to your system PATH
- **Verify & uninstall** — Check your installation or cleanly remove everything
- **Localized** — Auto-detects and uses your system language

## Download

Grab the latest binary for your OS from the [releases page](https://github.com/Cristiancastt/ADB-USB-Driver-Installer/releases/latest):

| Platform | File |
|---|---|
| Windows (x64) | `adb-installer-*-win-x64.zip` |
| Linux (x64) | `adb-installer-*-linux-x64.tar.gz` |
| Linux (ARM64) | `adb-installer-*-linux-arm64.tar.gz` |
| macOS (Intel) | `adb-installer-*-osx-x64.tar.gz` |
| macOS (Apple Silicon) | `adb-installer-*-osx-arm64.tar.gz` |

Extract it and run:

**Just run it** — launches the install wizard
```bash
./adb-installer
```

**Or be explicit**
```bash
./adb-installer install
```

**Check if everything's working**
```bash
./adb-installer verify
```

**Clean uninstall**
```bash
./adb-installer uninstall
```


## Building from source

You need the [.NET 10 SDK](https://dotnet.microsoft.com/download/dotnet/10.0).

```bash
git clone https://github.com/Cristiancastt/ADB-USB-Driver-Installer.git
cd ADB-USB-Driver-Installer
dotnet run --project CLI
```

Publish a self-contained binary:

```bash
dotnet publish CLI/CLI.csproj -c Release -r linux-x64 --self-contained -p:PublishSingleFile=true -o out/
```

Swap `linux-x64` for `win-x64`, `osx-arm64`, etc.

## Verifying releases

Every release is signed and checksummed. Don't trust random binaries from the internet — verify them.

### 1. Check the hash

Download the binary + SHA256SUMS from the release:

```bash
sha256sum -c SHA256SUMS
```

(on macOS: `shasum -a 256 -c SHA256SUMS`)

### 2. Verify the GPG signature

Import the project maintainer's public key:
KEY ID: `889328956A98A0E2BBB01D9D9BDCDF7FF4A84772`

```bash
gpg --keyserver keyserver.ubuntu.com --recv-keys <KEY_ID>
```

Verify the checksums file was signed by that key:

```bash
gpg --verify SHA256SUMS.asc SHA256SUMS
```

### 3. Inspect build provenance

Each platform also has a `.buildinfo.json` with the exact git commit, compiler version,
build command, timestamps, and a full dependency list with hashes. You can compare this
against what you'd get building from source yourself.

### How to add a translation

It's simple — we use `.resx` (resource) files:

1. **Copy an existing translation** from `CLI/Resources/` (e.g., `Strings.es.resx`)
2. **Translate the strings** to your language
3. **Rename** to match your language code (e.g., `Strings.ja.resx` for Japanese)
4. **Send a PR** with your translation

Each `.resx` file is just XML with key-value pairs. No need to be a developer — if you can translate text, you can help!
